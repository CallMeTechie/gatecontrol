# Pro Theme Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add a second, professional "Pro" theme to GateControl with Clean Corporate Light design (Royal Blue accent, Inter font, Stripe/Linear aesthetic) alongside the existing "default" theme, switchable per-user and via environment variable.

**Architecture:** The existing template system already supports themes via `config.theme.defaultTheme` (env `GC_DEFAULT_THEME`) and per-user `theme` column in the users table. Templates use `{% include theme + "/partials/..." %}` for dynamic includes. The new theme creates a complete `templates/pro/` directory with its own layout, partials, and pages that reuse the same JavaScript files (business logic) but apply a new CSS design system. One rendering fix is needed: `res.render()` must use `res.locals.theme` instead of `config.theme.defaultTheme` to enable per-user theme switching.

**Tech Stack:** Nunjucks templates, CSS (new `public/css/pro.css`), existing Express/Node.js backend

**Reference Mockups:** `/root/GC-Mockups/` (10 HTML mockup files showing the target design)

### Patterns to Follow

Every page template extends layout:
```nunjucks
{% extends theme + "/layout.njk" %}
{% block content %}
  {# page content #}
{% endblock %}
{% block scripts %}
  <script src="/js/pagename.js"></script>
{% endblock %}
```

**Nonce convention:** The default theme uses `nonce="{{ cspNonce }}"` ONLY on inline `<script>` and `<style>` blocks, NOT on `<script src="...">` tags (which are covered by CSP `'self'`). The Pro theme must follow the same pattern.

Standalone pages (login, error, route-auth-login) use their own full HTML.

All templates use the same variables/context as the default theme:
- `{{ t('key') }}` for i18n translations
- `{{ csrfToken }}`, `{{ cspNonce }}` for security
- `{{ activeNav }}` for sidebar highlighting
- `{{ license.hasFeature('x') }}` for feature gates
- `{{ peerCount }}`, `{{ routeCount }}`, etc. for badges
- `{{ user }}`, `{{ flash }}` for session data

## Task 2: Fix Render Path for Per-User Theme Switching

**Files:**
- Modify: `src/routes/index.js:127-143`
- Modify: `src/routes/auth.js:12-16`
- Modify: `src/app.js:136-169`

Currently, `res.render()` calls hardcode `config.theme.defaultTheme` in the template path argument, even though `res.locals.theme` is already correctly set per-user by `locals.js` (used in template includes). The render path must also use `res.locals.theme` for full per-user switching. Additionally, `src/routes/routeAuth.js:150` has a render call for the public route-auth-login page that also needs fixing.

- [ ] **Step 1: Fix page rendering in routes/index.js**

In `src/routes/index.js`, change the render call at line ~138 from:

```javascript
res.render(`${config.theme.defaultTheme}/pages/${template}.njk`, {
```

to:

```javascript
const theme = res.locals.theme || config.theme.defaultTheme;
res.render(`${theme}/pages/${template}.njk`, {
```

- [ ] **Step 2: Fix login page rendering in routes/auth.js**

In `src/routes/auth.js`, the `loginPage` method at line ~12 renders the login page. Change:


Note: For the login page, `res.locals.theme` may not be set (user not authenticated). The `|| config.theme.defaultTheme` fallback handles this.

- [ ] **Step 3: Fix error/404 rendering in app.js**
